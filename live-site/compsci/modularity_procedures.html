Chapter 14: Modularity & Procedures

Building complex systems using reusable, interrelated, and independent blocks of code.

In this chapter: 14.1 The Power of the "Black Box" 14.2 Interrelated Programs 14.3 Managing Complexity

When you first learn to code, you write programs that are 10 or 20 lines long, executing straight from top to bottom. But modern video games and apps have millions of lines of code. If a developer tried to read a million lines straight down, their brain would melt. To build massive software systems without losing our minds, we must master Modularity.

14.1 The Power of the "Black Box" Standard 9-12.AP.M.1

Imagine driving a car. You press the gas pedal, and the car moves forward. Do you need to know exactly how the fuel injector mixes gasoline and oxygen to create combustion? No! The engine is a "Black Box"—you give it an input (pressing the pedal), and it gives you an output (movement). Programming works the exact same way.

14.1.1 Defining Modularity (The DRY Principle)

Modularity is the design technique of breaking a large software program into smaller, self-contained units (modules) where each unit performs exactly one specific task.

The golden rule of software engineering is DRY: Don't Repeat Yourself. If you are building an online store, you need to calculate sales tax when the user views their cart, and again when they checkout, and again when you print their receipt. If you type the math formula for calculating tax three different times, you are violating the DRY principle. What happens if the tax rate changes next year? You have to hunt down and fix all three formulas. Instead, we use a Procedure.

14.1.2 Decomposition into Procedures

A Procedure (also known as a Function or a Method) is a named block of code that you write once, and then "Call" (trigger) as many times as you want. By decomposing a program into procedures, you turn 100 lines of repetitive math into a single, clean command: calculateTax().

14.1.3 Inputs and Outputs (Parameters)

For a procedure to be a true "Black Box," it needs to accept data, process it, and return a result. The data we send into the procedure is called a Parameter (or Argument).

Inputs (Parameters)

When you call calculateTax( 50 ), the number 50 is the input parameter. The procedure catches that number and uses it to do the math.

Outputs (Return)

Once the math is done, the procedure uses a Return command to shoot the final answer back to the main program (e.g., returning 54.00).

Step-by-Step: Writing a Custom Procedure in Sheets

You don't need to write Python to create a procedure. We can create a custom Named Function right inside Google Sheets!

  1. Open the Menu: In Google Sheets, click Data > Named functions.
  2. Create the Box: Click "Add new function." Give it a name like CALCULATE_TAX.
  3. Set the Parameters (Inputs): In the "Argument placeholders" box, type price and hit Enter. This tells the computer to expect one input.
  4. Write the Code: In the "Formula definition" box, type the math: =price + (price * 0.08). Click Save.
  5. Call the Procedure: Click on any empty cell in your sheet and type =CALCULATE_TAX(100). It will instantly return 108! You just built a reusable module!

14.2 Interrelated Programs Standard 9-12.AP.M.2

A single procedure is helpful, but true software engineering happens when multiple independent modules talk to each other to solve a massive problem.

14.2.1 Building Systems

A video game is not one program; it is a System of interrelated programs. The Movement module calculates where the player is standing. It then talks to the Graphics module, sending it the coordinates so the Graphics module knows where to draw the character on the screen. Because they are modular, the Graphics developer can update the character's 3D model without ever accidentally breaking the Movement developer's physics code.

14.2.2 Combinations of Data and Procedures

In Chapter 13, we learned about Arrays. Procedures and Arrays are a match made in heaven. You can create a procedure called findHighestScore(), and pass an entire Array of 10,000 scores into it as a single parameter. The procedure loops through the array, finds the winner, and returns the result. This keeps your main program perfectly clean.

14.2.3 Independent but Interrelated

For modules to work together safely, they must be independent. This means the "Login" module shouldn't care how the "User Profile" module stores its data. The Login module just verifies the password and says, "Hey User Profile, the password was correct. Load the screen for Alice." If they are too deeply tangled together (what programmers call "Spaghetti Code"), fixing a bug in the Login screen might accidentally delete Alice's profile picture!

14.3 Managing Complexity Standard 9-12.AP.M.1

As a system gets larger, a Data Architect must enforce strict rules to prevent the code from collapsing under its own weight.

14.3.1 Scope and Local Variables

To keep procedures truly independent, programming languages use a concept called Scope. Scope is the "Las Vegas Rule" of coding: What happens inside the procedure, stays inside the procedure.

The Soundproof Room

If you create a variable named totalCoins inside the calculateScore() procedure, that variable is Local to that specific procedure. It exists in a soundproof room. If you try to ask for totalCoins from outside the procedure, the computer will crash and say it has no idea what you are talking about. This is a good thing! It prevents other modules from accidentally overwriting your math.

14.3.2 Library Logic

The ultimate form of DRY (Don't Repeat Yourself) is realizing that someone else has probably already solved your problem. A Library is a collection of pre-written, highly tested modules created by other developers that you can download and plug directly into your own program.

If you are building an app that needs to find the distance between two GPS coordinates on a map, you don't need to invent complex spherical geometry math. You just download a "Map Math Library," and call their calculateDistance(lat1, lat2) procedure. Don't reinvent the wheel!

14.3.3 Troubleshooting Modules (Unit Testing)

When a massive game crashes, finding the bug can be terrifying. But if the game is highly modular, developers use Unit Testing. They unplug all the modules from each other and test them one by one in isolation. They feed a fake number into the `calculateScore` module. If it returns the correct math, they know that specific unit is safe, and the bug must be hiding in a different module.
