Unit 6: Software Engineering

Chapter 15: The Software Development Lifecycle

Planning, testing, and refining software for real human users.

In this chapter: 15.1 The Iterative Design Process 15.2 Accessibility & Inclusive Design 15.3 Collaborative Tools & Docs

Writing code is actually the easiest part of making an app. The hard part is figuring out what to build, ensuring it actually solves a problem, and making sure other people can use it without getting frustrated. Professional developers use a structured framework called the Software Development Lifecycle (SDLC) to guide a project from a rough idea to a polished final product.

15.1 The Iterative Design Process Standard 9-12.AP.PD.1

You should never try to build an entire app perfectly on the first try. Instead, software engineering is Iterative. You build a tiny piece, test it, fix it, and repeat the cycle until the product is complete.

15.1.1 The SDLC Cycle

Every major software project follows these five core stages:

1. Plan

Identify the problem and the audience.

2. Design

Create flowcharts and UI sketches.

3. Build

Write the actual algorithms and code.

4. Test

Find bugs and get user feedback.

5. Maintain

Release updates and patch security.

15.1.2 Building for an Audience

If you build an app that you think is brilliant, but nobody else can figure out how to use it, your app is a failure. You must identify exactly who will use your program. An educational game designed for a 5-year-old needs massive buttons, bright colors, and zero text. A financial app designed for an accountant needs dense data tables, keyboard shortcuts, and complex charts.

15.1.3 Incorporating User Feedback Standard 9-12.AP.PD.1

You cannot test your own app. Because you built it, you know exactly how it is supposed to work, so you will unconsciously avoid clicking the "wrong" buttons. To truly test software, you must conduct a User Interview (Beta Testing).

The Golden Rule of Beta Testing

When you give your prototype to a user, do not help them. Tell them the goal (e.g., "Try to buy a pair of shoes"), and then stand back and watch in silence. If they click the wrong button or get lost, do not correct them! Their failure is your fault, not theirs. It means your UI is confusing. Write down where they got stuck, and change your code based on what they actually did.

Activity 15.1.4: The Beta Test

Share your "Budget Tracker" or "Interactive Slide Dashboard" with a classmate. The "User" must try to break the program or find confusing parts without asking for help. The "Developer" must silently watch, record the feedback, and plan one specific fix.

15.2 Accessibility & Inclusive Design Standard 9-12.AP.PD.3

In the physical world, we build wheelchair ramps so everyone can access a building. In the digital world, we must build software that accommodates visual, hearing, and motor impairments.

15.2.1 Designing for Everyone

Accessibility (a11y) is the practice of designing products that are usable by people with disabilities. True inclusive design doesn't just help people with disabilities—it helps everyone.

  • High-Contrast Colors: Helps users with color blindness distinguish between buttons, but also helps users trying to read their phone in bright sunlight.
  • Keyboard Shortcuts & Tab Navigation: Crucial for users with motor impairments who cannot use a mouse, but also heavily used by "Power Users" who want to work faster.
  • Closed Captions: Essential for the deaf and hard of hearing, but also used by people watching videos on a loud, crowded bus.
Step-by-Step: Evaluating Accessibility

How do you know if your app is accessible? You must test it using the same tools that people with disabilities use.

  1. The Mouse Test: Unplug your mouse (or disable your trackpad). Can you navigate through your entire app and click all the buttons using ONLY the Tab, Arrow, and Enter keys?
  2. The Squint Test: Step back 10 feet from your monitor and squint. Are the primary buttons still obvious? Is the text still readable? If not, your contrast is too low or your fonts are too small.
  3. The Screen Reader Test: Turn on Narrator (Windows) or VoiceOver (Mac). Close your eyes. Can you understand what is happening on the screen based entirely on the audio feedback?

15.2.2 Refinement for Usability

Once you identify "Pain Points" during an accessibility audit, you must refine the interface. If an interface is cluttered with 50 tiny buttons, it is not accessible. A skilled designer simplifies the complex, guiding the user's eye to the most important action on the screen.

15.3 Collaborative Tools & Documentation Standards 9-12.AP.PD.4, 9-12.AP.PD.5

Software is rarely built by a single "lone wolf" programmer. It is built by highly coordinated teams. To avoid chaos, these teams must use specialized roles, version control systems, and rigorous documentation.

15.3.1 Team Roles in Tech

A successful tech team divides the labor into specific, specialized roles (Standard 9-12.AP.PD.4):

Project Manager

They do not write code. They talk to the clients, set the deadlines, organize the tasks, and ensure the team stays on track.

UX/UI Designer

They do not write logic algorithms. They interview users, create the visual sketches, and ensure the app is beautiful and accessible.

Software Engineer

They take the designer's sketches and the manager's requirements and write the actual procedures, loops, and data structures to make it work.

15.3.2 Version Control

If two programmers try to edit the same file at the same time, one will overwrite the other's work. To prevent this, developers use Version Control Systems (like Git and GitHub).

Version control takes a snapshot of the code every time a change is made. If a new update accidentally breaks the entire app, the team doesn't panic—they just click a button to instantly "roll back" the code to the version from yesterday. You already practice a basic form of this when you use Google Drive's "Version History" to recover a deleted essay!

15.3.3 Documenting Design Decisions Standard 9-12.AP.PD.5

Writing code is only half the job; explaining why you wrote it is the other half. Documentation is the process of using text, graphics, and presentations to explain how a program works.

If you leave the company and a new programmer takes over your app, they need to read your Documentation to understand why you chose to use an Array instead of a Linked List, or why you designed the database the way you did. Good documentation ensures that software can survive long after its original creator moves on.
