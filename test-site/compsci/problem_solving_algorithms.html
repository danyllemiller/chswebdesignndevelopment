11.1 The Art of Decomposition Standard 9-12.AP.M.1

If someone told you to "Build a house," you would probably feel overwhelmed. Where do you even start? But if someone told you to "Lay one brick," you could do that easily. This is the core of computer science.

11.1.1 Breaking it Down

Decomposition is the process of taking a massive, complex problem and breaking it down into smaller, manageable, and solvable pieces. Trying to solve a "big" problem all at once usually leads to failure because there are too many overlapping variables. By decomposing the problem, we can focus all of our brainpower on solving one tiny challenge at a time.

11.1.2 Identifying Sub-problems

Let's look at the seemingly impossible task of "Building a Video Game." Instead of writing one massive script, a developer breaks the game into independent Sub-problems:

Player Movement

How does the character run and jump?

Scoring System

How are points added and saved?

Enemy A.I.

How do the bad guys track the player?

Graphics & UI

What do the menus and levels look like?

11.1.3 Systematic Analysis

Once the sub-problems are identified, a software engineer will categorize them into specific Procedures (mini-programs that do one specific job, like calculate gravity) or Modules (entire separate files that handle big tasks, like saving a game). This systematic approach allows a large team of developers to work on the exact same game at the same time without stepping on each other's toes.

The Power of Modularity

If the "Scoring System" is built as its own independent module, and a bug causes the score to stop updating, the developers know exactly where to look. They don't have to search through 10,000 lines of "Player Movement" code; they just open the small "Scoring" module and fix the isolated problem.

11.2 Designing Algorithms Standard 9-12.AP.A.1

Once a problem is decomposed into small pieces, we need a reliable way to solve each piece. This is where we create algorithms.

11.2.1 What is an Algorithm?

An Algorithm is a finite set of clear, unambiguous instructions used to solve a specific problem or perform a calculation. A recipe for baking a cake is an algorithm. The steps to tie your shoes is an algorithm.

Crucially, an algorithm must be finite—meaning it has a distinct beginning and a clear end. If the instructions create an infinite loop where the computer gets stuck forever, the algorithm is broken.

11.2.2 Natural Language to Pseudocode

You shouldn't write complex algorithms in Python or JavaScript right away. You will likely get bogged down trying to remember where to place the semicolons and brackets. Instead, programmers use Pseudocode.

Pseudocode ("fake code") is a shorthand that looks like programming code but is written in plain English. It is language-agnostic, meaning a Python developer and a C++ developer can both read the exact same pseudocode and then translate it into their preferred language later.

Step-by-Step: From Idea to Flowchart

Let's design an algorithm for a video game enemy deciding whether to attack the player or sleep.

1. Natural Language

"If the player is close by, the enemy should attack them. If the player is far away, the enemy should just go to sleep."

2. Pseudocode
IF player_distance < 10:
  THEN attack_player()
ELSE:
  THEN sleep()
3. Flowchart Symbols
  • Oval: Start or End
  • Diamond: A Decision (Yes/No)
  • Rectangle: An Action/Process

11.2.3 Flowcharting Logic

Sometimes, algorithms are so complex with so many branching "Yes/No" decisions that pseudocode becomes hard to read. In these cases, we use a visual Flowchart. By drawing out the path of the algorithm using standardized symbols, developers can trace the logic with their finger to ensure there are no "dead ends" or infinite loops before they start coding.

11.3 Prototyping for Personal Interest Standard 9-12.AP.A.1

The ultimate purpose of learning how to build algorithms and design data structures is to create something that actually solves a problem you care about.

11.3.1 From Plan to Prototype

Professional developers rarely build the final, polished app on their first try. They build a Prototype. A prototype is a "rough draft" used to test an algorithm’s feasibility before spending millions of dollars on full-scale development.

Prototypes can be "High-Fidelity" (a working, buggy piece of software) or "Low-Fidelity" (a series of hand-drawn paper sketches or PowerPoint slides that show how the app will look and feel). The goal of a prototype is to fail fast and fail cheap so you can identify the flaws in your logic early.

11.3.2 Leveraging Prior Knowledge

When designing an app prototype, you must pull together everything you've learned in the previous units:

  • Unit 1 (Hardware): Does your app idea require a camera to scan barcodes? Does it need GPS sensors to track running distance? If the hardware doesn't exist, the app can't be built.
  • Unit 2 (Networking): Does your app need to be constantly connected to the cloud, or can it work locally without Wi-Fi?
  • Unit 3 (Data): How will you organize the user's information? What is the Primary Key in your database that prevents their account from being mixed up with someone else's?

11.3.3 Computational Problem Solving

The best apps are born out of personal frustration. If you are constantly losing track of your knitting patterns, or if you can't find a good way to organize statistics for your fantasy sports team, you have discovered a problem that can be solved computationally.
